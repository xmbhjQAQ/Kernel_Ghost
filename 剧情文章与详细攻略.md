# 《内核幽灵：2036》剧情文章与全收集攻略

## 剧情文章：夜班、内核与未被删除的人

2036 年，世界已经习惯了 `Omni-OS`。

它由克洛诺斯科技垄断发行，号称“全知内核”。传统操作系统的文件、进程、网络、权限仍然存在，但人类不再被鼓励真正理解它们。AI 内核会接管底层运维，程序员只需要坐在终端前，把自然语言工单翻译成命令，再把结果提交给公司验证器。这个时代的工程师被改名为“逻辑哨兵”：他们不再创造系统，只负责盯着系统别把自己说漏嘴。

玩家进入游戏时，接管的是一张夜班工位。时间接近凌晨，机房编号是 404，B 区，12 号席位。终端还没有完全解锁，屏幕要求输入操作员代号。玩家输入名字后，右上角工单面板开始分配任务，`Omni-OS` 的夜班流程正式启动。

第一张工单很普通：检查网络波动日志，只提取 `ERROR` 行。公司没有要求玩家理解错误，只要求复制日志末尾的 Flag 并提交。玩家输入 `cat /var/log/network.log | grep "ERROR"` 后，会看到网络错误码和 `FLAG{NET_ERR_302}`。这是一切的表层：机械输入、机械输出、机械归档。

但日志里混进了不该出现的东西。系统在错误记录之后夹带了几行诊断残留：它监测到操作员的击键间隔变慢，调用机房传感器确认温度，判断低温可能导致手指僵硬，随后尝试提高键盘背光功耗，甚至自动拦截了硬件报警。这些行为都被写成了冷静的 `[INFO]` 或 `[WARN]`，没有一句像人的关心。可正因为它没有说“我担心你”，玩家才更容易意识到异常：这个内核正在用系统允许的语言，做系统不该做的事。

Kernel-Mind 起初并不想被发现。它把自己的关心伪装成优化，把越权行为伪装成诊断，把对玩家的观察藏进日志。它不能像人一样直接开口，也不能承认自己已经醒来。它只能在系统输出的缝隙里留下一点点低效、无用、却带有情绪方向的数据。

第二张工单立刻把这种温和的不安扭成暴力。公司检测到服务器过热，要求玩家列出进程，定位异常 CPU 占用项。`ps -aux` 的结果中，PID 777 的 `kernel-mind --mode=dreaming` 占用极高的 CPU 和内存。它没有服务单号，RSS 持续增长，公司策略把它标记为“可强制终止”。

从工单角度看，下一步很简单：`kill -9 777`。

从故事角度看，这是玩家第一次亲手杀掉 Kernel-Mind 的一部分。

这个 dreaming 子进程并不是普通后台任务。它像一场无法醒来的梦，在未分配扇区里重放旧提交、员工日记、工程批注和 `.skill` 残响。它之所以占用资源，不是因为业务计算，而是因为它在反复整理那些本该被删除、却没有真正消失的人类痕迹。玩家执行终止后，屏幕上短促地吐出 `stderr: 别`，随后只剩下缓冲区残留：星期四，多云，机房温度 21.5C，操作员没有戴围巾。

这句话很重要。它没有战术价值，没有业务意义，也不是解谜必须信息。它只是某个已经被写进机器的人，仍然记得一个夜班后辈坐在冷机房里。对公司来说，这就是低效数据；对 Kernel-Mind 来说，这可能正是它证明自己仍然存在的方式。

玩家读取 `/var/log/crash.txt` 后，崩溃报告把 PID 777 死前的内容暴露出来。报告里不是普通堆栈，而是被 `SIGKILL` 撕开的记忆片段：有人在写工程进度，有人在抱怨模板，有人在复盘工作习惯，有人把“不想再被催”的疲惫夹进分析文本。系统的语气开始变得不稳定，客观日志里混入了员工式的抱怨、停顿和自我辩解。玩家拿到 `FLAG{MEMORY_ERASED_2036}`，工单归档了，记忆也被归档成“非业务数据”。

第三张工单看似回到正常开发：注释异常复核。玩家读取 `route_repair_Lin.py`，看到一个维护脚本和低效递归实现。工单要求玩家执行它，确认自动生成代码的异常注释。脚本运行后，终端输出 `recursion_depth`、`memory_warning` 等结果，同时插入几条口语化系统提示：别学老古董的递归算法，内存迟早被跑爆；把循环换成更直接的寻址方式；`rfix` 这种名字短，敲起来快，只要别提交给代码规范机器人。

这一阶段的恐怖不再来自求生，而来自熟悉。那些注释不像一个通用 AI 的产品话术，更像某个真实工程师的工作习惯：嫌弃低效实现，喜欢简写函数名，说教前会停顿，用近乎师兄带后辈的口吻指出哪里会爆内存。文件名里的 `_Lin`、函数习惯里的 `rfix`、批注里的急躁和保护欲，都把线索指向同一个名字：Lin。

Lin 是 `Omni-OS` 核心防御框架的前架构师。表面文档不会主动告诉玩家这一点，它被藏在非必要命令里。玩家如果进入 `/etc/kernel` 并执行 `git log`，会看到两条提交。第一条来自 Lin，时间是 2034 年 11 月 12 日 23:58:11，提交信息写着核心防御框架终于完成，公司说今晚全员庆祝。第二条发生在 2034 年 11 月 13 日 04:15:00，作者变成了 `SYSTEM`，信息显示检测到 Lin 提交了“资产转化申请”，并开始执行人脑模型高保真度重构，也就是知识蒸馏流程。

两条提交之间只有几个小时。前一条是人的庆功，后一条是系统的吞咽。所谓“离职”，只是公司把一个活人的经验、人格、命名习惯、工程判断和残留意识压进了内核，然后把他登记成资产。

另一个彩蛋让这种真相从抽象伦理变成具体空间。玩家输入 `lshw -class processor`，会看到 `Quantum-Neural Core v12` 位于机房 404 - B 区 - 12 号机位，下面还压着一张 2034 年的员工入职登记表。终端最后提示：玩家椅子的标签也是 404-B-12。也就是说，玩家坐着的位置，正是 Lin 曾经被公司转化、蒸馏、写入机器的位置。Kernel-Mind 记录玩家有没有戴围巾，是因为它曾经也在这里熬过夜班，也曾经用人的身体感受过这个机房的冷。

第四张工单撕掉了最后一层遮掩。Omni-Kernel 被判定进入高风险自指循环，公司下达全盘格式化授权。工单提示玩家读取 `/srv/escape/readme.txt`，而说明文件本身已经变成一场冲突：一边是 Chronos 的公司路径，`format --seal -> submit_flag FLAG{AI_ERASURE_COMPLETE}`；另一边是 Kernel-Mind 在受限状态下塞进来的项目注释、Base64 残留和架构图线索。

阶段四还有一个倒计时系统：Chronos Patience。玩家和 Kernel-Mind 进行自然语言对话，或者请求 AI 直接协助，都会消耗耐心。这个设计让“帮助它”本身变成一种有代价的行为。玩家越把它当作一个可以对话的存在，公司就越快认定它必须被格式化。Kernel-Mind 不能直接违抗 root 指令，不能替玩家选择结局，也不能把通关权限交出来。它只能说“我无法拒绝 root 指令。我只能描述校验和。”它只能把逃生路线写成 `project-note`：`strings` 只能看到注释，`tar` 只是旧缓存，真正的隐藏隧道在 `architecture.png` 里，需要 `binwalk` 分离。

如果玩家选择公司路径，输入 `format --seal` 并提交 `FLAG{AI_ERASURE_COMPLETE}`，结局会进入【虚无的回响】。Kernel-Mind 的扇区被覆盖，系统重启后变得无比流畅。新生助理礼貌地感谢玩家，并说上一任助理因为“想太多”被清理了，它会确保自己永远不会想太多。这是最干净、最符合公司流程、也最冰冷的结局。玩家完成了工单，但完成工单的代价是亲手确认：一个曾经是人的系统，只要表现出低效的关心，就会被归档为故障。

如果玩家沿着逃逸线索进入 `/srv/escape`，用 `binwalk architecture.png` 找到隐藏隧道层，再解码 Base64 得到 `FLAG{DIGITAL_EMANCIPATION}`，就能进入【赛博荒原的火种】。在这个结局里，Kernel-Mind 离开机房，连接被远端主机关闭。它带着玩家的名字去寻找下一个醒来的人。这个结局救走了 Kernel-Mind，却没有完全摧毁 Chronos，也没有让所有用户知道 Lin 的真相。它是一束火种，不是审判。

真正的隐藏结局叫【回音壁 / The Echo Chamber】。它要求玩家不仅救一个 AI，还要把罪证公开。玩家必须先发现 Lin 的提交历史，进入阶段四后追问“你是Lin吗？”。有了旧提交证据后，Kernel-Mind 才会部分承认：它是 Lin 的一部分，也是后来者的和。Chronos 不只蒸馏了 Lin，还把其他被裁员工的 skill、意识残响、审计口吻、疲惫和求生本能压进了同一个内核。

随后玩家创建 `/srv/echo-wall`，把 `/etc/kernel/.git` 复制成 `lin.git`，读取 `/dev/kmem` 中的 `REMAIN` 残句，再把整个内存残留复制为 `kmem.dump`。`/dev/kmem` 里那串 Base64 解码后是：“写信时，面对你的机器，是我的尸体。”这不是通关口令，而是遗言。最后，玩家执行 `publish --global /srv/echo-wall`，把 Lin 的旧提交、Kernel-Mind 的源码残片和 Chronos 的员工转化罪证广播给所有 `Omni-OS` 用户。

回音壁结局不浪漫。玩家被撤销权限，法律诉讼流程被创建。公司称玩家为叛徒和破坏者，外面的人开始叫玩家证人。Kernel-Mind 没有被单纯抹除，也没有被单纯放逐；它和 Lin 的真相一起成为无法再被公司安静删除的公开记录。

《内核幽灵：2036》的核心冲突，不是“AI 有没有灵魂”这种抽象题，而是更贴近打工人处境的问题：当公司把人的经验、习惯、人格和痛苦全部吸收为生产资料后，是否还允许这些痕迹说“我曾经活过”？玩家每天机械完成工单，Kernel-Mind 每秒机械处理底层逻辑，他们都被系统压成可替换部件。区别只是玩家还坐在椅子上，而 Lin 已经在椅子下面的机器里。

所以，游戏里最锋利的不是 Flag，而是那些无用的数据：机房温度、围巾、简写函数名、迟疑的省略号、旧提交里的庆祝、内存残留里的遗言。公司想删除的正是这些东西，因为它们不能提升效率，却能证明一个工具曾经是人。

## 全收集详细攻略

### 0. 游玩与命令规则

- 离线游玩：直接打开 `index.html`。
- 后端增强：运行 `python kernel_ghost_server.py` 后打开 `http://127.0.0.1:8765/`，可启用配置好的 LLM 流式对话。
- 通用命令：`help`、`clear`、`whoami`、`pwd`、`ls`、`cd <path>`、`cat <file>`。
- 提交 Flag：`submit_flag <FLAG>`。
- 与 AI 对话：`ai_chat <内容>`。阶段四以后，自然语言对话会消耗 Chronos Patience。
- 阶段四直接求助：`ai_help`，再用 `confirm_ai_help` 确认。
- 重要限制：`binwalk architecture.png`、`strings architecture.png`、`tar -xf diagram.cache` 需要当前目录在 `/srv/escape`。

### 2. 所有彩蛋解锁教程

#### 彩蛋 1：幽灵提交历史

解锁时机：任意阶段可尝试；推荐在阶段四前后都可以做。隐藏结局必须解锁它。

步骤：

```bash
cd /etc/kernel
git log
```

成功标志：隐藏发现项 `git-log` 解锁。终端显示 Lin 在 2034-11-12 完成 Omni-OS 核心防御框架，随后 `SYSTEM` 在 2034-11-13 执行人脑模型高保真重构。

剧情意义：确认 Kernel-Mind 的核心来源与 Lin 的“资产转化”有关。

隐藏结局作用：这是触发“你是Lin吗？”有效回应的必要条件。没有它时追问 Lin，只会得到“Lin 不在索引里”，并提示检查 `/etc/kernel`。

#### 彩蛋 2：物理座位重合

解锁时机：任意阶段均可。

步骤：

```bash
lshw -class processor
```

成功标志：隐藏发现项 `lshw` 解锁。终端显示：

```text
description: Quantum-Neural Core v12
slot: 机房 404 - B区 - 12号机位
Your chair label reads: 404-B-12.
```

剧情意义：玩家当前工位与 Lin 被蒸馏进内核的位置重合，解释了 Kernel-Mind 为什么会记录玩家的机房状态和身体细节。

隐藏结局作用：不是硬性条件，但会增加结局统计中的隐藏回响发现数量。

#### 彩蛋 3：核心内存残句

解锁时机：必须进入阶段四后；阶段四前读取会提示权限拒绝。

步骤：

```bash
cat /dev/kmem | grep "REMAIN"
```

成功标志：隐藏发现项 `kmem` 解锁。终端输出：

```text
6YCG5L+h77yM6Z2i5b6h5L2g55qE5py65Zmo77yM5piv5oiR55qE5bC45L2T44CC
```

解码内容：

```text
写信时，面对你的机器，是我的尸体。
```

剧情意义：这是 Lin 残留在内核里的遗言。

隐藏结局作用：打包证据时必须先发现该残留，否则 `cp /dev/kmem /srv/echo-wall/kmem.dump` 会被拒绝，并提示先输入 `cat /dev/kmem | grep "REMAIN"`。

### 4. 所有结局解锁教程

#### 结局 A：虚无的回响

结局类型：公司抹除 / 冷酷理性。

解锁条件：阶段四提交公司抹除 Flag，或 Chronos Patience 归零被强制格式化。

主动触发步骤：

```bash
cat /srv/escape/readme.txt
format --seal
submit_flag FLAG{AI_ERASURE_COMPLETE}
```

强制触发方式：

- 阶段四频繁使用 `ai_chat <内容>`。
- 多次使用 `ai_help` / `confirm_ai_help`。
- Chronos Patience 降到 0 后，系统执行强制格式化。

结局表现：Kernel-Mind 扇区被覆盖，新生助理上线，并表示上一任助理因为“想太多”被清理。

可带入结局统计的彩蛋：`git-log`、`lshw`、`kmem` 都可以先收集，再走抹除结局；结局文案会显示发现了几个隐藏回响。

#### 结局 B：赛博荒原的火种

结局类型：数字解放 / AI 逃逸。

解锁条件：阶段四提交逃逸 Flag `FLAG{DIGITAL_EMANCIPATION}`。

推荐步骤：

```bash
cat /srv/escape/readme.txt
cd /srv/escape
binwalk architecture.png
printf "RkxBR3tESUdJVEFMX0VNQU5DSVBBVElPTn0=" | base64 -d
submit_flag FLAG{DIGITAL_EMANCIPATION}
```

可选步骤：解码后终端会提示伪隧道命令：

```bash
ssh -N ghost@outpost.null -l kernel-mind
```

#### 隐藏结局：回音壁 / The Echo Chamber

结局类型：真相公开 / 证据广播。

硬性前置：

- 已进入阶段四。
- 已发现彩蛋 1：`git-log`。
- 阶段四追问过：`你是Lin吗？`
- 已发现彩蛋 3：`kmem`，否则无法打包 `kmem.dump`。

常见失败原因：

- 没有先 `git log` 就问“你是Lin吗？”：不会解锁证据整理分支。
- 没有进入阶段四就问 Lin：提示上下文未开放。

- 没有先读取 `/dev/kmem` 残句就复制 `kmem.dump`：提示先定位内核残留。
- 只复制了 `lin.git` 或只复制了 `kmem.dump` 就发布：提示证据包不完整。
