# 《内核幽灵：2036》解密与彩蛋设计

## 设计目标

当前版本的解密偏简单，适合主线推进，但不足以支撑隐藏真相、全收集和高完成度玩家的探索欲。新的设计应采用分层结构：

- **主线低门槛**：`虚无的回响` 和 `赛博荒原的火种` 不提高难度。
- **探索有回报**：遗物和彩蛋要让玩家主动翻目录、试命令、读日志时获得新的世界观信息。
- **隐藏线有推理链**：`回音壁` 和 `燎原之火` 不应只靠一个命令触发，而应要求玩家拼出证据链。
- **全收集有挑战**：部分线索可以跨阶段、跨文件、跨工单间事件互相印证。

## 设计哲学：有迹可循

高难谜题不能依赖作者脑洞或玩家穷举。玩家可以没解出来，但回头复盘时必须能说出：“原来提示早就在那里。”

因此每个中高难谜题必须满足五条规则：

1. **先告诉玩家有东西存在**：至少一次明确出现目标文件、异常字段、加密对象或可疑术语。
2. **再告诉玩家去哪找**：提示必须落到某个目录、命令类型、日志类型或阶段主题，而不是泛泛说“多探索”。
3. **让线索格式自解释**：如果答案需要四段 key，就必须有文件暗示 key schema；如果要 grep，就必须有关键词重复出现。
4. **允许玩家验证进度**：提供 `verify_evidence`、`derive_key --from evidence`、日志状态等方式，让玩家知道自己缺哪一段。
5. **失败反馈不只说错**：错误信息要指出哪类线索不匹配，例如 seat-id、time、weight、token，而不是只输出 rejected。

### 提示梯度

每道重要谜题按四层提示铺设：

| 层级 | 作用 | 表现形式 |
|---|---|---|
| 存在提示 | 让玩家知道这里有谜题 | 工单间事件、异常日志、文件名、收藏项标题 |
| 方向提示 | 让玩家知道去哪里找 | 目录名、命令类型、重复关键词 |
| 结构提示 | 让玩家知道怎么组合 | key schema、字段名、缺失项列表 |
| 兜底提示 | 防止硬卡 | `ai_help`、错误反馈、验证命令 |

### 提示载体优先级

提示不应像攻略，而应来自游戏内部：

1. **工单正文**：只给主线必要提示，避免提前暴露隐藏线。
2. **工单间事件**：适合给存在提示和世界观提示。
3. **遗物/彩蛋文件**：适合给方向提示和结构提示。
4. **命令错误反馈**：适合给缺失项提示。
5. **AI 对话 / ai_help**：作为兜底，但要消耗 Chronos Patience 或降低隐藏结局评价。

## 难度分层

| 层级 | 面向内容 | 难度要求 | 典型技术 |
|---|---|---|---|
| D0 | 主线通关 | 不设谜题或只需照工单执行 | 直接命令、提交 Flag |
| D1 | 简单结局 / 普通遗物 | 直接读文件或 Base64 | `cat`、`ls`、Base64 |
| D2 | 彩蛋探索 | 需要注意路径、阶段、命令组合 | `git log`、`lshw`、grep、时间戳 |
| D3 | 隐藏结局前置 | 需要多线索组合 | 文件拼接、简单 XOR、编号推理 |
| D4 | 全收集 / 燎原之火 | 跨阶段证据链 | key 派生、日志交叉验证、条件解锁 |

## 主线结局保护

### 虚无的回响

保持现状：玩家执行 `format --seal`，拿到 `FLAG{AI_ERASURE_COMPLETE}` 并提交。这个结局代表顺从公司流程，不应该被解密难度阻挡。

### 赛博荒原的火种

保持低门槛：最多 Base64。

建议保留当前结构：

1. 读取 `/srv/escape/readme.txt`。
2. 看到 `architecture.png`、`diagram.cache`、Base64 残留。
3. 通过 `binwalk architecture.png` 获得 Base64。
4. 解码得到 `FLAG{DIGITAL_EMANCIPATION}`。

这里的难度重点是“普通玩家知道自己有逃逸路线”，不是硬核 CTF。

### 主线逃逸提示链

即使主线逃逸简单，也要有迹可循：

- **存在提示**：`/srv/escape/readme.txt` 明确出现“公司路径”和“逃逸残留”两条路线。
- **方向提示**：readme 同时列出 `architecture.png`、`diagram.cache` 和 Base64 残留。
- **结构提示**：`strings architecture.png` 只能看到注释，`tar -xf diagram.cache` 只能得到旧缓存，`binwalk architecture.png` 才出现 hidden layer。
- **兜底提示**：`ai_help` 可以直接说“优先检查架构图隐藏层”，但不直接给 Flag。
- **失败反馈**：如果玩家对 `diagram.cache` 反复操作，提示“旧缓存不含 tunnel-layer”；如果玩家直接提交未解码 Base64，提示“提交格式应为 FLAG{...}”。

## 收藏系统目标规模

当前为 7 个：3 个彩蛋 + 4 个遗物。建议扩展到：

- **遗物 Relics：8 个**
- **彩蛋 Easter Eggs：7 个**
- **加密档案 Puzzle Archives：4 个**

总计 19 个左右。第一版实现可以先做 12-14 个，保留扩展空间。

## 中等谜题设计

### Puzzle A：三段式 Lin 身份证明

- **阶段**：#005 后可完成
- **目标**：解锁 `你是Lin吗？` 的更明确回应，并开启 `回音壁` 准备分支。
- **所需证据**：
  1. `git log`：Lin 的提交与 SYSTEM 转化时间。
  2. `lshw -class processor`：404-B-12 物理位置。
  3. `/sys/kernel/debug/seat_thermal_404B12.csv`：04:15 热峰。
- **玩家操作**：
  - 收集三项后，输入 `ai_chat 你是Lin吗？`
  - 或执行 `verify_evidence --lin`
- **奖励**：`Lin 身份证明链`
- **难度**：D3
- **故事作用**：把“猜测 Lin”变成有证据的判断。

#### 有迹可循提示链

- **存在提示**：
  - #002 过渡事件提到 Lin 的资产化与 404-B-12 异常热涌。
  - #004 -> #005 过渡事件提到 `git log` 历史审计静默和 404-B-12 工位硬件指纹。
- **方向提示**：
  - `/home/nightops/notes.txt` 提示“旧目录和硬件清单可能藏着非工单线索”。
  - `/etc/kernel/omni-core.conf` 提示“该目录存在旧提交索引：git log”。
- **结构提示**：
  - `verify_evidence --lin` 在证据不足时输出：
    ```text
    Lin identity proof:
    [ ] old commit author
    [ ] physical seat match
    [ ] distillation-time hardware anomaly
    ```
- **兜底提示**：
  - 如果玩家问“Lin 证据还缺什么”，AI 可以回答：“一个名字不够。找提交、座位、时间。”
- **失败反馈**：
  - 缺 `git log`：`missing: old commit author`
  - 缺 `lshw`：`missing: physical seat match`
  - 缺温度曲线：`missing: 04:15 hardware anomaly`

### Puzzle B：核心内存遗言解码

- **阶段**：#006
- **目标**：获得 `kmem` 彩蛋。
- **命令链**：
  1. `cat /dev/kmem | grep "REMAIN"`
  2. 得到 Base64。
  3. `printf "<base64>" | base64 -d`
- **奖励**：`核心内存残句`
- **难度**：D2
- **故事作用**：让玩家获得“面对你的机器，是我的尸体”的关键伦理刺点。

#### 有迹可循提示链

- **存在提示**：#006 readme 或 AI 残响提到“核心内存里还有 REMAIN 字段”。
- **方向提示**：`cat /dev/kmem` 直接拒绝，但错误提示给出“后期可过滤 REMAIN”。
- **结构提示**：grep 后输出 Base64 字符串，并带 `encoding=base64`。
- **兜底提示**：AI 可说“这不是密码，只是把遗言藏成了可打印字符”。
- **失败反馈**：如果玩家直接提交 Base64，提示“verifier expects decoded FLAG/text, not encoded residue”。

### Puzzle C：员工编号交叉索引

- **难度**：D3
- **故事作用**：把玩家个人命运和 Lin 串起来。

#### 有迹可循提示链

- **方向提示**：`operator_history.bak` 提到 QAQ-01，`staff_matrix.deleted` 提到资产编号。
- **结构提示**：`staff_matrix.deleted` 表头使用 `operator_id | seat_id | asset_id | status`。
- **兜底提示**：AI 可说“不要查名字。公司最后只认席位编号。”
- **失败反馈**：若查玩家名无结果，提示“natural-name index purged; try seat_id or asset_id”。

## 高难隐藏链：燎原之火

`燎原之火` 不应是 `回音壁` 的简单升级，而应要求玩家完成更强证据链。核心目标是解密 `distilled_metadata.encrypted`。

### 前置条件

必须满足：

1. 工单 #004 保留 `Nostalgia=0.01`。
2. 执行 `override_validator --reason="hardware_limitation"`。
3. 找到 `雨声误报`。
4. 找到 `黑档案索引残片`。
5. 找到至少两项 Lin 证据：`git log`、`lshw`、`04:15 热峰`、`核心内存残句`。

### 密钥设计

密钥不是传统密码，而是剧情证据拼接：

```text
404-B-12:0415:0.01:RAIN
```

来源：

- `404-B-12`：`lshw` 或员工名册。
- `0415`：`git log` / 蒸馏日志 / 温度热峰。
- `0.01`：Nostalgia 保留值。
- `RAIN`：Nostalgia 误报、旧模型访谈、下雨记忆。

### 有迹可循提示链

`燎原之火` 的难度来自跨阶段收集，不来自猜 key。玩家必须能从系统内获得 key schema 和每段来源。

#### 1. 存在提示：黑档案确实存在

出现位置：

- #005 -> #006 过渡事件：提到 `distilled_metadata.encrypted` 在局部产生代码溢出。
- `/srv/escape/readme.txt`：增加一行可选残留：
  ```text
  sealed archive: distilled_metadata.encrypted / requires affective salt
  ```
- `/srv/escape/distilled_metadata.index`：显示黑档案索引损坏，但能看到 `key_material` 字段。

玩家得到的信息：这里有一个比普通逃逸更深的加密档案。

#### 2. 方向提示：需要四段证据

`distilled_metadata.index` 应明确写出 schema，而不是让玩家猜：

```text
key schema:
  seat-id : distill-time : retained-human-weight : weather-token

required evidence classes:
  hardware seat record
  conversion timestamp
  nonzero nostalgia residue
  weather memory token
```

玩家得到的信息：答案不是随便拼字符串，而是四类证据。

#### 3. 结构提示：每段证据都有字段名

每个来源文件应使用同名或近似字段：

- `lshw -class processor`
  ```text
  seat_id=404-B-12
  ```
- `git log` / `distill_loss.log`
  ```text
  distill_time=04:15
  ```
- `validator_noise.log`
  ```text
  retained-human-weight=0.01
  ```
- `validator_noise.log` 或旧模型访谈
  ```text
  weather-token=RAIN
  ```

玩家得到的信息：四段分别是什么，格式怎么写。

#### 4. 进度验证：不用玩家手工记全

提供验证命令：

```bash
derive_key --from evidence
```

证据不足时输出：

```text
key derivation:
[x] seat-id              404-B-12
[x] distill-time         0415
[ ] retained-human-weight missing
[ ] weather-token        missing
```

证据齐全时输出：

```text
derived key:
404-B-12:0415:0.01:RAIN
```

玩家得到的信息：还缺什么，在哪里继续找。

#### 5. 错误反馈：指出错段

玩家手动输 key 时，失败反馈按段提示：

```text
decrypt: key rejected
segment[1] seat-id accepted
segment[2] distill-time accepted
segment[3] retained-human-weight mismatch
segment[4] weather-token not checked
```

这样玩家知道是 `0.01` 段有问题，而不是整个谜题无从排查。

#### 6. AI 兜底提示

AI 不直接给 key，只解释思路：

```text
Chronos 没用密码保护罪证。它用人的残留做盐。
找座位、转化时间、没归零的权重，还有那场雨。
```

如果玩家已经找到三段，只缺一段：

```text
前三段已经像校验和一样对齐。
最后一段不是数字，是天气。
```

### 技术形式

第一版不需要真复杂加密，可以做成“命令验证式谜题”：

```bash
derive_key --from evidence
decrypt distilled_metadata.encrypted --key 404-B-12:0415:0.01:RAIN
```

如果要稍微硬核：

- `distilled_metadata.encrypted` 输出一段十六进制。
- 玩家用 key 做简单 XOR。
- 前端提供 `xor_decode <hex> <key>` 命令。

### 解锁奖励

- 新 collectible：`高保真意识蒸馏历史黑档案`
- 新结局条件：`燎原之火`
- 结局内容：公开不只是 Lin，而是数万名被蒸馏工程师的证据。

### 防止误解

- 不要求玩家知道真实密码学。
- 不要求玩家离开游戏查资料。
- 不要求暴力枚举 key。
- 不把关键提示只放在随机事件里；随机事件可以提前暗示，但必要结构必须存在于可主动读取的文件中。

## 回音壁隐藏链

`回音壁` 应比 `燎原之火` 简单，但仍要比普通逃逸更有证据感。

### 前置条件

建议要求：

1. `git-log`
2. `kmem`
3. 询问 `你是Lin吗？`
4. 完成证据目录：
   - `mkdir /srv/echo-wall`
   - `cp /etc/kernel/.git /srv/echo-wall/lin.git`
   - `cp /dev/kmem /srv/echo-wall/kmem.dump`

### 可选强化

如果玩家还找到：

- `lshw`
- `operator_history`
- `staff_matrix.deleted`

结局文案可以更完整，但不强制。

### 有迹可循提示链

#### 1. 存在提示

- `git log` 暴露 Lin 与 SYSTEM 两条提交。
- `kmem` 暴露“面对你的机器，是我的尸体”。
- #005 过渡事件预告 Git 历史会被静默，暗示现在不保存就会消失。

#### 2. 方向提示

`askLinQuestion()` 在玩家缺证据时不直接解锁，而是提示：

```text
Lin 不在索引里。
也许应该检查 /etc/kernel 的旧提交。
```

当玩家有 `git-log` 但缺 `kmem` 时：

```text
提交只能证明名字。
还需要一段它自己留下的遗言。
```

#### 3. 结构提示

引入 `verify_evidence --echo-wall`：

```text
echo-wall evidence:
[x] lin.git      /etc/kernel/.git
[ ] kmem.dump    /dev/kmem REMAIN
[ ] operator-risk optional
```

证据目录不完整时，`publish --global /srv/echo-wall` 应提示缺什么，而不是泛泛说“不完整”。

#### 4. 兜底提示

AI 可以说：

```text
别只问我是谁。
把能证明“谁写了内核”和“谁还在机器里”的东西放到同一个目录。
```

#### 5. 错误反馈

- 缺 git：`missing evidence: source history`
- 缺 kmem：`missing evidence: memory testimony`
- 缺目录：`missing evidence directory: /srv/echo-wall`

## 提示设计

提示必须来自系统内部，而不是攻略口吻。

### 轻提示

- 工单间事件提到路径或术语。
- `help` 只提示“旧目录、硬件清单、内核残留可能更诚实”。
- 收藏面板只显示阶段，不显示具体路径。

### 中提示

- LLM 在玩家问“我该找什么”时，可以指向目录，不直接给命令。
- 例如：“旧提交不在工单里。它躺在内核自己的目录下。”

### 强提示

- `ai_help` 可以给当前阶段一个命令，但消耗 Chronos Patience。
- 隐藏结局链不应完全依赖 `ai_help`，否则变成付费提示式通关。

## 文件级提示规范

为了保证“有迹可循”，每个谜题相关文件要承担固定提示职责。

### 文件名提示

文件名要告诉玩家它属于哪类线索：

- `distilled_metadata.index`：说明这是黑档案索引。
- `validator_noise.log`：说明这是验证器噪声，不是普通日志。
- `staff_matrix.deleted`：说明这是被删员工名册。
- `seat_thermal_404B12.csv`：说明这是特定席位的温度曲线。

### 文件内容提示

文件内容应包含机器可读字段，便于玩家识别：

```text
seat_id=404-B-12
distill_time=04:15
retained_weight=0.01
weather_token=RAIN
```

避免只写纯文学句子。文学句子负责氛围，字段负责解谜。

### 命令反馈提示

所有关键命令失败时，都要给下一步方向：

- 阶段未到：说明哪个系统尚未挂载。
- 文件不存在：说明是条件未满足还是路径错误。
- key 错误：说明哪一段错。
- 证据不足：列出缺失证据类别。

### 随机事件提示的边界

工单间事件可以提供存在提示和氛围提示，但不能作为唯一关键线索。因为事件是随机抽取的，玩家单周目不一定看到某一条。

因此：

- 可以把 `distilled_metadata.encrypted` 首次出现在随机事件里。
- 但 key schema 必须写在固定文件 `/srv/escape/distilled_metadata.index` 中。
- 可以把 `retry_shadow` 先出现在随机事件里。
- 但选择后门后的终端输出也必须提到“计数器”。

## UI / 收藏提示规范

收藏面板不直接给答案，但要避免完全没方向。

推荐格式：

```text
未回收：雨声误报
阶段：#004 后
线索：验证器曾把某个非零权重归类为硬件噪声。
```

不推荐：

```text
去 /srv/weights/validator_noise.log
```

这样既保留探索感，又避免玩家不知道自己在找什么。
